### Как работать с новым бэклогом и отчётами

- backlog/CONTEXT-ALL.md — сводный контекст по всем 14 доменам (рамка, текущее покрытие, GAPы). Начинать работу отсюда.
- backlog/sources.md — опорные источники (локальные пути + внешние при появлении). Добавляйте сюда новые нормативные/технические ссылки.
- backlog/weights.md — веса доменов (сейчас равные) и методика SP (Fibonacci) + текущая уверенность (SCI). Используйте при пересчёте прогресса/приоритизации.
- backlog/gaps.md — все SPEC/INFO GAPы и отсылки к YAML-патчам (payout schedule, transfer REST, audit, /metrics и т.д.). Перед разработкой сверяться тут и сначала закрывать спеки.
- backlog/epics/EPIC-D*-*.md — эпики по доменам: бизнес-ценность, KPI, Security/Observability/CI-CD/Prod, риски, зависимости, Linked Stories, GAPы. Использовать для планирования спринтов.
- backlog/stories/US-D*-*.md — пользовательские истории с Gherkin-AC, Security/Observability/CI-CD/Prod, Negative cases, SP и приоритетом. Берём задачи отсюда; если API/Event нет — пометка GAP.
- backlog/matrix.csv — таблица связей Domain→Epic→Story (Role, API, Events, Data, Security, Tests, Ops, SP) для трекинга покрытия и приоритизации.
- reports/progress/PROGRESS-20250202.md — отчёт прогресса (DMI, BC/EVI/SCI, GAP-лист).
- reports/progress/plan-next-week.md — приоритетные действия на неделю (закрытие GAP спек/аудита/метрик/e2e и т.д.).

Как работать:
1) Перед задачей: найти историю в backlog/stories, сверить с эпиком и GAPами. Если API/Event отсутствует — сначала применить YAML-патч из gaps.md, обновить спеки/SDK.
2) Планирование: использовать matrix.csv для выбора stories по домену/роли/риску; опираться на SP и веса доменов (weights.md).
3) Прогресс: обновлять DoR/DoD и артефакты тестов/репортов, затем пересчитывать отчёт в reports/progress.
4) Security/Observability/CI-CD/Prod: секции в историях обязательны к закрытию, иначе DoD=0.
5) Новые источники — в sources.md; новые GAPы — в gaps.md с патчами.

Это обеспечивает прослеживаемость: домен → эпик → история → спецификация → реализация/тесты.

Текущий процент готовности можно оценить лишь ориентировочно, т.к. DoD по основным флоу не подтверждены тестами/артефактами и сервисы не связаны. По данным прогресс-отчёта (BC≈0.43, EVI=0, SCI=0.45) итоговая готовность: ~19% (с учётом низкой уверенности).

Основания:

1. Спеки покрыты частично (BC≈0.43), но интеграции не проверены, DLT не подключён, события/отчёты/платёжные статусы имеют GAP.
2. Авт/аут базово есть; фронт-порталы работают на моках, без связки с бэком/ДЛТ.
3. CI/CD и observability — WIP, нет /metrics у gateway, нет алертов и полноценных пайплайнов.
4. Тестовые артефакты почти отсутствуют (DoD=0).

При улучшении связности сервисов, закрытии GAP и появлении тестовых отчётов процент можно пересчитать.
